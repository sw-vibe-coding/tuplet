



How can I have named tuples in Rust?

You’re effectively asking for:

“How do I get tuple-like positional access and named fields in Rust?”

This touches on data modeling choices, not just syntax. Rust deliberately separates:

Tuples → positional, lightweight, anonymous

Structs → named, explicit, extensible

There is no built-in “named tuple” type like Python’s namedtuple or OCaml’s labeled tuples—but you can achieve equivalent (and often better) designs using a few patterns.

🧠 Design Considerations
Before jumping into syntax, think about what you actually want:

Requirement	Best Rust Construct
Named fields, readable	struct
Tuple-like compactness	tuple struct
Pattern matching + names	struct
Interop / lightweight	tuple or tuple struct
Python-style namedtuple	struct + destructuring
Rust pushes you toward clarity over brevity, so most “named tuple” use cases map cleanly to struct.

✅ Option 1 — Regular Struct (Idiomatic Rust)
This is the canonical solution.

struct Point {
    x: i32,
    y: i32,
}

let p = Point { x: 10, y: 20 };

println!("{}", p.x);
Why this is usually best
Self-documenting

Works with pattern matching

Easy to extend later

Zero runtime cost

Destructuring (tuple-like feel)
let Point { x, y } = p;
✅ Option 2 — Tuple Struct (Hybrid)
If you want tuple syntax + type name, use a tuple struct:

struct Point(i32, i32);

let p = Point(10, 20);
println!("{}", p.0);
Tradeoffs
✔ lightweight

✔ newtype-friendly

✘ no field names (.0, .1 only)

✅ Option 3 — Struct + impl Accessors (Encapsulation)
If you want both names and controlled access:

struct Point(i32, i32);

impl Point {
    fn x(&self) -> i32 { self.0 }
    fn y(&self) -> i32 { self.1 }
}

let p = Point(10, 20);
println!("{}", p.x());
This gives:

tuple storage

named API

✅ Option 4 — Pattern Matching Alias (Closest to Named Tuple Feel)
You can destructure inline to simulate named tuples:

let (x, y) = (10, 20);
println!("{x}, {y}");
Or:

let point = (10, 20);
let (x, y) = point;
This is the closest to “named tuple variables” but:

names exist only locally

not part of the type

⚙️ Option 5 — Macros / Crates (Python-style namedtuple)
There are crates that mimic named tuples:

named_tuple

tuple_struct

derive_more

Example (conceptually):

named_tuple!(Point { x: i32, y: i32 });

let p = Point { x: 1, y: 2 };
Tradeoffs
✔ concise

✘ adds dependency

✘ not idiomatic Rust

✘ less flexible than structs

🧩 Option 6 — Type Alias for Tuple (Not Truly Named)
type Point = (i32, i32);
Usage:

let p: Point = (10, 20);
But:

❌ no field names

❌ no safety improvement

🧠 Architectural Guidance
Given your broader work (compilers, VMs, IRs, etc.), here’s how to think about it:

Use struct when:
Modeling AST nodes

IR representations

VM state

Anything evolving over time

Use tuple structs when:
Newtype wrappers (e.g., Address(u32))

Performance-critical tight representations

Use tuples when:
Temporary return values

Small local transformations

⚖️ Comparison Summary
Feature	Tuple	Tuple Struct	Struct
Named fields	❌	❌	✅
Lightweight	✅	✅	✅
Readability	❌	⚠️	✅
Extensible	❌	❌	✅
Pattern matching	✅	✅	✅
🚀 Practical Recommendation
For your use cases (compilers, IR, MLPL, COR24 tooling):

Default to struct unless you have a very specific reason not to.

If you want “named tuple ergonomics,” combine:

struct

short field names

named semantics

👍 Bottom line
Rust doesn’t have “named tuples” because:

struct is the named tuple—just more powerful.

Rust does not have automatic tuple-splicing in function calls, so this will not work directly:

But Rust can model the idea cleanly with named tuple-like structs and a trait that converts different “point descriptions” into the canonical four plotting arguments.

Rust version: tuple variables as “argument bundles”
#[derive(Debug, Clone, Copy)]
struct Coord2 {
    x: i32,
    y: i32,
}

#[derive(Debug, Clone, Copy)]
struct Coord3 {
    x: i32,
    y: i32,
    z: i32,
}

#[derive(Debug, Clone, Copy)]
enum Color {
    Red,
    Green,
    Blue,
}

#[derive(Debug, Clone, Copy)]
struct Transparency(f32);

impl Transparency {
    fn percent(value: f32) -> Self {
        Self(value / 100.0)
    }
}

#[derive(Debug, Clone, Copy)]
struct Point4 {
    x: i32,
    y: i32,
    color: Color,
    transparency: Transparency,
}
Now define a trait meaning:

“This thing can be expanded into the arguments needed by plot.”

trait IntoPlotArgs {
    fn into_plot_args(self) -> Point4;
}
Implement it for a full point:

impl IntoPlotArgs for Point4 {
    fn into_plot_args(self) -> Point4 {
        self
    }
}
And for a coordinate plus extra values:

impl IntoPlotArgs for (Coord2, Color, Transparency) {
    fn into_plot_args(self) -> Point4 {
        let (coord, color, transparency) = self;

        Point4 {
            x: coord.x,
            y: coord.y,
            color,
            transparency,
        }
    }
}
Now plot accepts anything that can become plotting arguments:

fn plot(args: impl IntoPlotArgs) -> bool {
    let Point4 {
        x,
        y,
        color,
        transparency,
    } = args.into_plot_args();

    println!(
        "plot x={x}, y={y}, color={color:?}, transparency={}",
        transparency.0
    );

    true
}
Usage:

fn main() {
    let coord2 = Coord2 { x: 3, y: 9 };

    // Equivalent to: plot⎛coord₂ Red 50%⎠

    println!("success = {success}");

In the original language idea, this:

plot⎛coord₂ Red 50%⎠
means coord₂ contributes two stack values.

In Rust, values do not automatically expand into multiple arguments. Instead, we usually make the grouping explicit:

plot((coord2, Color::Red, Transparency::percent(50.0)));
That outer tuple is the “argument bundle.”

A programming language designer claims that no programming language has named tuples.  What does this mean?

Let’s unpack the claim carefully, because it’s not literally true, but it can be true depending on how the designer is defining terms.

🧠 Restating the claim
“No programming language has named tuples.”

This usually means:

What people call “named tuples” are actually something else (typically structs/records), not true tuples with names.

So the argument hinges on what counts as a tuple.

🧩 Step 1 — What is a “tuple” (strict sense)?
In a strict, type-theoretic / PL design sense:

A tuple is:

Ordered

Positional

Anonymous field access

Identified by arity and types only

Example (Rust / many languages):

let t: (i32, i32) = (3, 9);
Access:

t.0
t.1
Key property:

The meaning of each element is determined by position, not name.

🧩 Step 2 — What is a “named tuple”?
In many languages (Python, C#, Swift, etc.), a “named tuple” looks like:

Point(x=3, y=9)
or:

(int x, int y) p = (3, 9);
These give you:

Positional structure

Named field access (p.x, p.y)

🧨 The designer’s argument
The claim is usually:

The moment you give tuple elements names, it is no longer a tuple — it becomes a record/struct.

Because:

1. Names change identity semantics
Compare:

(i32, i32)
vs:

{x: i32, y: i32}
In many type systems:

(i32, i32) ≠ (y: i32, x: i32) (if names matter)

But {x: i32, y: i32} = {y: i32, x: i32} (order doesn’t matter)

So:

Named fields imply record semantics, not tuple semantics.

2. Access mode changes
Tuple:

t.0
Named tuple:

p.x
That’s no longer positional—it’s field-based access.

3. Evolution behavior differs
Tuples:

Fragile to change (add/remove/reorder breaks everything)

Records:

Can evolve (add fields, reorder, defaults, etc.)

So:

Named tuples behave like lightweight records, not tuples.

⚖️ So what languages call “named tuples”?
Python — collections.namedtuple
Point = namedtuple('Point', ['x', 'y'])
Internally:

It's basically a tuple subclass + attribute access

But semantically:

Acts like a record

C#
(int x, int y) p = (3, 9);
Names:

Are mostly compile-time sugar

Can be erased or ignored

So:

It’s still fundamentally a tuple

Names are optional metadata

Rust
No named tuples.

You must use:

struct Point { x: i32, y: i32 }
Rust’s stance is effectively:

If you want names, use a struct.

Swift
let p = (x: 3, y: 9)
But:

Type identity is still positional

Names are partially cosmetic

🧠 The deeper philosophical point
The designer is likely asserting:

There are only two real categories:

Tuples → positional, anonymous

Records/structs → named, semantic

And “named tuples” are:

either

tuples with cosmetic labels

or records pretending to be tuples

🧩 Mapping to your earlier language ideas
Your system:

⎛x y⎠
is interesting because:

It is a tuple

But you’re associating names with positions

And allowing destructuring:

a , b ⟵ coord₂
So your design is closer to:

tuples with semantic labels that don’t change positional identity

That’s actually rare.

⚖️ Summary
The claim:

“No programming language has named tuples”

means:

In a strict sense, you must choose:

positional tuples

or named records

but not both simultaneously without changing semantics.

👍 Practical takeaway
Most languages that “have named tuples” are:

either adding labels to tuples (cosmetic)

or providing lightweight structs

Truly combining:

positional semantics

named access

interchangeable usage

is non-trivial and uncommon

Trait approach
#[derive(Clone, Copy)]
struct Coord2 {
    x: i32,
    y: i32,

#[derive(Clone, Copy)]
enum Color {
    Red,
    Green,
    Blue,
}

#[derive(Clone, Copy)]
struct Transparency(f32);

struct PlotArgs {
    x: i32,
    y: i32,
    color: Color,
    transparency: Transparency,
}

trait IntoPlotArgs {
    fn into_plot_args(self) -> PlotArgs;
}

impl IntoPlotArgs for PlotArgs {
    fn into_plot_args(self) -> PlotArgs {
        self
    }
}

impl IntoPlotArgs for (Coord2, Color, Transparency) {
    fn into_plot_args(self) -> PlotArgs {
        let (coord, color, transparency) = self;

        PlotArgs {
            x: coord.x,
            y: coord.y,
            color,
            transparency,
        }
    }
}

fn plot(args: impl IntoPlotArgs) -> bool {
    let args = args.into_plot_args();

    println!(
        "plot {}, {}, {:?}, {}",
        args.x,
        args.y,
        args.color,
        args.transparency.0
    );

    true
}

The apparent meaning is:

  n : integer
  e : integer

Power returns:
  p : integer
and the implementation is roughly:

if e is positive:
    start with 1
    loop e times multiplying by n
A corresponding conventional pseudocode reading:

Power(n : ℤ, e : ℤ) -> p : ℤ =
    p ← 1
    repeat e times:
        p ← p × n
    return p
Or in Rust-ish form:

fn power(n: i64, e: i64) -> i64 {
    assert!(e >= 0);

    let mut p = 1;

    for _ in 0..e {
        p *= n;
    }

    p
}
For your language design, this suggests the syntax has:

Syntax	Likely role
• Power	function/verb definition
(n : ℤ e : ℤ)	named typed input tuple
→ (p : ℤ)	named typed output tuple
←	definition/body introducer
⎧ ⎨ ⎩	guarded alternatives / structured body
iff e is positive	guard / condition
n (×) →	repeated reduction / multiplication pipeline
The interesting connection to the earlier tuple discussion is that both the input side and output side are already named tuples:

(n : ℤ  e : ℤ) → (p : ℤ)
So a compiler could treat this as:

Power : ⎛n : ℤ, e : ℤ⎠ ───‣ ⎛p : ℤ⎠
and lower it to a stack effect:

This fits the OCaml-front-end / Forth-like-IR design very naturally.
